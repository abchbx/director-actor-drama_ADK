```python
STORM_INTERVAL = 8          # Default scenes between auto-triggers (D-11)
MAX_TRIGGER_HISTORY = 10    # Max trigger history entries to keep

OVERLAP_THRESHOLD = 0.6     # Keyword overlap ratio threshold (D-13)
CONFLICT_KEYWORD_MAP = {
    "隐藏": "secret_revealed", "秘密": "secret_revealed",

    "新角色": "new_character", "登场": "new_character",
}
```

3. **discover_perspectives_prompt(state, focus_area="")** function:
   - Extracts from state: `storm.get("perspectives", [])` names, `conflict_engine` tension/active_conflicts/consecutive_low_tension, `plot_threads` dormant threads, `actors` arc_progress, `scenes` last 3 scene summaries, `storm.get("outline", {})`
   - Builds structured prompt with sections: 已有视角, 当前张力状态, 活跃冲突, 休眠线索, 角色弧线进展, 近期场景, 世界观设定
   - Includes core instructions from D-03: "基于当前剧情状态，发现 1-2 个尚未被探索的新视角或新角度", "新视角应能引入新的冲突可能性或扩展故事世界的边界", "不要重复已有视角", "新视角必须与已发生事件一致，是扩展而非推翻"
   - Instructs LLM to return JSON array: `[{"name": "...", "description": "...", "questions": ["..."]}]`
   - Returns the complete prompt string

4. **check_keyword_overlap(new_name, existing_names)** function:
   - Extract 2-4 char Chinese segments from new_name (sliding window)
   - Filter out segments that are entirely in VIRTUAL_WORDS
   - Extract same from each existing_name
   - Calculate overlap ratio: `len(intersection) / len(new_keywords)` if new_keywords else 0
   - Return `{"overlap_ratio": float, "overlap_warning": str | None, "overlapping_with": list[str]}`
   - If overlap_ratio > OVERLAP_THRESHOLD: set overlap_warning like `新视角'{new_name}'与已有视角'{matching_name}'可能重叠`
   - Follow D-13/D-14/D-15: overlap does NOT block generation, only warns

5. **suggest_conflict_types(perspective_description)** function:
   - Check perspective_description against CONFLICT_KEYWORD_MAP keys
   - Return list of matching conflict type strings (deduplicated)
   - If no matches, return empty list (not an error)

6. **parse_llm_perspectives(response_text)** function:
   - Try to extract JSON array from response_text (handle ```json blocks)
   - Parse each item into `{name, description, questions}` format
   - Validate: name must be non-empty string, description must be non-empty string, questions defaults to []
   - Return list of validated perspective dicts
   - On parse failure: return empty list (graceful degradation)

7. **_init_dynamic_storm_defaults()** function (per D-27/D-28):
   - Return dict: `{"scenes_since_last_storm": 0, "trigger_history": [], "discovered_perspectives": []}`

8. **update_dynamic_storm_state(state, trigger_type, focus_area, perspectives_found, new_perspectives)** function (per D-30/D-31):
   - Get or init `state["dynamic_storm"]` via `_init_dynamic_storm_defaults()`
   - If trigger_type is non-empty: reset `scenes_since_last_storm = 0`, append to `trigger_history`: `{"scene": state.get("current_scene", 0), "trigger_type": trigger_type, "focus_area": focus_area, "perspectives_found": perspectives_found}`
   - Trim `trigger_history` to `MAX_TRIGGER_HISTORY`
   - If new_perspectives provided: extend `discovered_perspectives`
   - Return updated `dynamic_storm` dict (caller persists to state)

Follow conflict_engine.py patterns exactly: pure functions, state: dict in, dict out, bilingual docstrings.
