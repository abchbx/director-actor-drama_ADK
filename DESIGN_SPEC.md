# DESIGN_SPEC.md

## 概述

这是一个导演-演员动态 Multi-Agent 戏剧创作系统。导演 Agent 采用 **STORM（Synthesis of Topic Outlines through Retrieval and Multi-perspective Question Asking）框架**，通过多视角发现、深度研究、大纲合成、场景执行四个阶段进行戏剧创作。导演会根据剧本需要动态创建演员 SubAgent，每个演员拥有独立的记忆系统和认知边界。系统采用半自动模式——在关键节点需要用户输入命令来推进剧情。系统支持通过命令启动、保存进度、退出后恢复进度，并将完整剧本保存为 Markdown 文档。

## 核心架构（STORM + A2A 多 Agent）

- **STORM 路由器（StormRouter）**：作为 root_agent（BaseAgent），根据戏剧状态将消息路由到对应的 STORM 阶段 Agent
- **发现者 Agent（Discoverer）**：STORM Phase 1，从多视角生成探索性问题
- **研究者 Agent（Researcher）**：STORM Phase 2，对每个视角进行深度研究
- **大纲合成者 Agent（Outliner）**：STORM Phase 3，将多视角研究结果融合为戏剧大纲
- **导演 Agent（Director）**：STORM Phase 4，负责场景执行、旁白叙述、动态创建演员、协调场景推进
- **演员 Agent（Actor）**：动态创建的独立 A2A 服务，每个演员运行在独立端口，拥有自己的会话和记忆，认知边界通过物理隔离保证
- **A2A 通信**：导演通过 `RemoteA2aAgent` 与演员通信，确保真正的多 Agent 隔离
- **命令系统**：用户通过命令控制流程，非全自动
- **持久化系统**：保存/恢复整个系统状态（剧本、演员、场景进度、STORM 数据）
- **剧本输出**：实时保存为 Markdown 文档（含 STORM 大纲信息）

## STORM 框架四阶段

### Phase 1 - 发现阶段（Discovery）
- 从多个视角（主角、反派、旁观者、伦理、命运）生成探索性问题
- 每个视角包含 3+ 个引导问题
- 产出：多视角列表 + 引导问题

### Phase 2 - 研究阶段（Research）
- 对每个视角进行深度研究
- 产出：角色原型、冲突模式、情感曲线、意象符号、跨视角联系
- 所有视角研究完成后自动过渡到大纲合成阶段

### Phase 3 - 大纲合成阶段（Outline Synthesis）
- 将多视角研究结果辩证融合为结构化戏剧大纲
- 产出：四幕结构（起承转合）+ 核心张力 + 主题层次 + 视角整合方案
- 合成完成后创建演员 A2A 服务

### Phase 4 - 场景执行阶段（Directing）
- 基于大纲执行戏剧演出
- 旁白叙述、角色调度、情感管理、剧本记录
- 在关键时刻回溯 STORM 发现的多视角洞察

## 示例用例

1. 用户输入主题"太空探险" → STORM 发现阶段：生成5个视角（指挥官、叛军、殖民地平民、伦理、命运）→ 研究阶段：每个视角深度挖掘 → 大纲合成：四幕结构 + 核心张力 → 创建"指挥官"、"工程师"、"医生"演员 → 用户 /next 推进场景 → 导演旁白衔接 → /save 保存 → 最终 /export 导出完整剧本
2. 用户输入主题"校园恋爱" → STORM 发现6个视角 → 研究发现"旁观者"和"主角"视角的深层矛盾 → 大纲合成体现这种矛盾 → 导演创建"男主"、"女主"、"闺蜜" → 演员间独立记忆（闺蜜不知道男主内心独白）→ 用户 /action 指令改变剧情走向
3. 用户 /save 保存后退出 → 下次 /load 恢复到之前场景（包括 STORM 数据）→ 继续推进
4. 用户 /action "突然停电" → 导演根据指令调整剧本 → 演员按新情境行动
5. 剧本完成 → /export 生成完整 Markdown 格式剧本（含 STORM 大纲、旁白、对白、场景描述）

## 工具需求

### STORM 框架工具（v1 新增）
- `storm_discover_perspectives`：从多视角探索戏剧主题（STORM Phase 1）
- `storm_ask_perspective_questions`：为特定视角生成深入问题（STORM Phase 2）
- `storm_research_perspective`：深度研究特定视角（STORM Phase 2）
- `storm_synthesize_outline`：将多视角研究合成为戏剧大纲（STORM Phase 3）

### 核心工具（沿用 v0）
- `create_actor`：动态创建演员 A2A 服务，设定角色名、性格、背景、认知边界，启动独立端口
- `actor_speak`：获取演员 A2A 连接信息，将对话委派给演员独立 Agent
- `update_script`：更新剧本内容（场景、旁白、对白）
- `save_progress`：保存当前进度（剧本、演员状态、场景信息、STORM 数据）
- `load_progress`：恢复之前保存的进度（含 STORM 数据）
- `export_script`：将完整剧本导出为 Markdown 文件（含 STORM 大纲）
- `director_narrate`：导演作为旁白描述场景过渡
- `advance_scene`：推进到下一个场景
- `start_drama`：初始化戏剧

## 命令系统

| 命令 | 描述 |
|------|------|
| `/start <主题>` | 开始新剧作，提供主题 |
| `/next` | 推进到下一个场景/情节 |
| `/action <描述>` | 用户注入一个事件或动作 |
| `/speak <角色名> <内容>` | 让用户直接替角色说话 |
| `/save` | 保存当前进度 |
| `/load [存档名]` | 加载之前保存的进度 |
| `/export` | 导出完整剧本为 Markdown |
| `/cast` | 查看当前所有演员及其状态 |
| `/status` | 查看当前场景和进度 |
| `/quit` | 退出系统（自动保存） |

## 约束与安全规则

- 演员的认知边界必须严格：演员 A 不知道演员 B 的内心活动（除非通过对话获知）
- 导演不能代替演员做决定，只作为旁白和引导
- 用户命令优先级最高，导演必须遵循用户指令
- 保存进度时必须完整序列化所有状态（包括 STORM 数据）
- 演员不允许"跳出角色"——必须严格按角色设定行动
- 动态创建的演员数量上限为 10 个
- STORM 各阶段必须按序执行：发现 → 研究 → 大纲 → 执行
- 模型使用 OpenAI 兼容接口（LiteLLM），配置为 openai/claude-sonnet-4-6

## 成功标准

- STORM 发现阶段能生成至少 5 个不同视角的探索方向
- 研究阶段能为每个视角产出具体的戏剧素材（角色原型、冲突模式等）
- 大纲合成能将多视角融合为有层次感的戏剧结构
- 导演能根据 STORM 大纲执行戏剧演出
- 演员能根据独立记忆和认知边界进行角色扮演
- 用户命令能正确控制剧情流程
- 保存/恢复功能可靠工作（包括 STORM 数据）
- 最终生成的 Markdown 剧本格式完整、可读（含 STORM 大纲）
- 系统支持至少 5 轮以上的连续剧情推进

## 边界情况

1. 用户输入不明确：STORM 发现阶段应主动询问澄清
2. 演员之间冲突：导演旁白介入调解或引导
3. 保存文件损坏：提供错误提示和部分恢复
4. 演员数量过多：导演合并角色或限制创建
5. 用户长时间不输入：系统等待，不自动推进
6. STORM 阶段跳跃：StormRouter 严格按状态路由，不允许跳过阶段
7. 加载 v0 保存文件：storm 字段为空，路由到 discoverer 重新开始 STORM 流程
